# Dejavu — два механизма памяти: cue-побудитель и долговременный store

> Документ-определение (задача #260, 2026-06-28, ревизия 2). Назначение — закрепить **устоявшиеся
> имена** для двух механизмов памяти агента, чтобы в коде, обсуждениях и трекере все называли их
> одинаково. Это словарь + модель ролей, а не дизайн механики: слои L0–L5 описывает
> `ASSOCIATIVE-MEMORY.md`, хранилище — `PLAN.md`.
>
> **Ревизия 2 (важно):** переосмыслен быстрый слой. Ранее он назывался «рабочая память /
> `flashMemory`» и определялся как «лог того, что происходит сейчас» (`activation_log`). **Это было
> неверно.** Его суть — не хранить текущее, а **нести сигналы-побудители**: дистиллят триггеров,
> который пробуждает обращение к долговременному store. Это фаза-1 (familiarity) двухфазности
> `ASSOCIATIVE-MEMORY.md §10.2`. Имя изменено на `cueMemory`; `activation_log` отнесён к телеметрии
> write-пути, а не к быстрому механизму.

---

## TL;DR

| | **Cue-память (побудитель)** | **Долговременный store** |
|---|---|---|
| Код-имя (API) | **`cueMemory`** (хэндл `cues`) | **`factStore`** (хэндл `facts`) |
| Русское имя | **«зацепки»** (альт: наводки, маяки) | **«свод»** (альт: копилка, факты) |
| Эргономика | `cues.match(context)` · «зацепка сработала» | `facts.get(cue)` · «достал из свода» |
| Психо-канон | familiarity / узнавание (фаза 1) | long-term memory (LTM), фаза 2 recollection |
| Роль | **побуждение**: «стоит ли смотреть» | **содержание**: «что именно» |
| Что несёт | дистиллят триггеров (cue + темы) | отобранные факты + связи + сила следа |
| Горизонт | индекс персистентен; взвод — на ход | межсессионный, персистентный |
| Носитель | `fact_cues` + SQLite-зеркало read-пути + флаг знакомости (L3) | `facts` + `fact_cues` + `fact_links` + `fact_embeddings` |
| Связь | **проекция** из `factStore` (factStore → cue) | source of truth |

**Одной фразой:** `cueMemory` — это дешёвый дистиллят триггеров, **спроецированный из** `factStore`;
он сопоставляется с живым контекстом каждый ход и при совпадении **взводит сигнал-побудитель**,
толкающий агента прочитать `factStore`. `cueMemory` отвечает «когда/стоит ли смотреть», `factStore` —
«что именно». Память становится **push**, а не pull, именно благодаря `cueMemory`.

---

## 1. Cue-память (`cueMemory`, фаза-1 familiarity)

**Определение.** Дешёвый, быстро-сопоставимый **дистиллят триггеров**, спроецированный из
`factStore`. Его задача — не хранить события, а **пробуждать**: сопоставить cue текущего контекста
(L0) со своим набором триггеров и при совпадении **взвести сигнал-побудитель** («тут есть
релевантная память, иди посмотри»), который толкает агента к чтению `factStore`. Это активный фронт
пассивного store — то, что превращает память из pull в push.

**Что несёт.**
- Символьные триггеры (`fact_cues`): имена контейнеров, порты, verbs, фрагменты путей, сигнатуры
  ошибок, темы/проекты — дискретные, мгновенно-матчуемые зацепки (ядро надёжного env-recall).
- (Приоритет 2) семантические якоря для нечёткого узнавания.
- На выходе — **флаг знакомости** (L3): короткая low-authority строка «⚡ память по теме: …»,
  почти бессодержательная, её роль — побудить, а не рассказать.

**Свойства.**
- *Горизонт.* Индекс триггеров персистентен (зеркалит `factStore`); **взвод** — эфемерен, на один
  ход (с habituation: повторно в сессии не дёргать).
- *Стоимость.* Без LLM/сети: символьное совпадение по плоскому/индексированному набору, десятки мс
  (`ASSOCIATIVE-MEMORY.md §3-L1.1`). Дёшево ровно потому, что не несёт содержимого.
- *Носитель.* `fact_cues` + локальное SQLite-зеркало read-пути + доставляемый флаг (L3).
  **В MVP сегодня** — поле `cues` в `facts.jsonl` (триггеры прицеплены к факту) + перебор в
  `dejavu-push.sh`; флаг знакомости — строка `⚡ память среды по теме: …`.
- *Чем НЕ является.* Не `activation_log` (это телеметрия write-пути, см. §3). Не «рабочая память /
  состояние сейчас». Не контекстное окно LLM.

## 2. Долговременный store (`factStore`, LTM, фаза-2 recollection)

**Определение.** Персистентное, межсессионное хранилище **отобранных** фактов с триггерами,
связями, эмбеддингами и силой следа — содержательная глубина, которую достаёт recollection после
того, как `cueMemory` взвёл побуждение.

**Что хранит.** `facts` (+ `strength` по Эббингаузу, `last_activated_at`, `activation_count`,
`salience`), `fact_cues` (символьный индекс — он же источник проекции в `cueMemory`), `fact_links`
(граф, `weight`, `co_activation_count`), `fact_embeddings`.

**Свойства.**
- *Горизонт.* Постоянно, межсессионно. System of record — Postgres `brain`; read-path дублируется
  в SQLite-зеркало (то самое, что обслуживает `cueMemory`).
- *Запись.* Дорогая, **salience-gated**: только значимое (новизна, поправки, повторы пользователя,
  env-открытия, разрешённые ошибки). Наполняется **консолидацией** (см. §3), не прямой записью.
- *Чтение.* `facts.get(cue)` — recollection: символьный exact + семантика + spreading по `fact_links`
  (L1), гейт L2, доставка L3. Плюс осознанный pull `memory_search` поверх того же store.
- *MVP сегодня.* Вырожден до плоского `facts.jsonl` (`id`/`cues`/`text`), правится руками.

## 3. Связь: три потока, не один

Раньше формулировка «быстрый питает долгосрочный» смешивала разные вещи. Честная модель — **три
отдельных потока**:

```
(A) ПРОЕКЦИЯ        factStore ──extract cues──► cueMemory
    factStore отдаёт триггеры каждого факта в дешёвый матчуемый индекс.
    Направление порождения: factStore → cueMemory.

(B) ПОБУЖДЕНИЕ      live context ──match──► cueMemory ──fire──► флаг (L3) ──► facts.get → recollection
    Фаза-1 (familiarity) взводит, фаза-2 (recollection) достаёт глубину. Это read-путь push-памяти.

(C) КОНСОЛИДАЦИЯ    сессионные сигналы (activation_log) ──L4 (salience-gate) / L5 (sleep)──► factStore
    Write-путь: наполнение и обслуживание store. ОТДЕЛЬНЫЙ механизм, НЕ cueMemory.
```

- **`cueMemory` исходит из `factStore`** (поток A) — проекция триггеров. «Один из другого»
  сохраняется, но направление: `factStore → cueMemory` (а не «быстрый → питает долгосрочный»).
- **Назначение `cueMemory` — разбудить `factStore`** (поток B), а не наполнить его.
- **Наполнение `factStore`** — это **консолидация** (поток C, L4+L5) из телеметрии сессии
  (`activation_log`: что всплыло, было ли использовано; самый ценный сигнал — повтор пользователя =
  push не сработал). Это write-процесс, его нельзя путать с быстрым механизмом.
- Происхождение факта из породившего эпизода фиксируется `fact_links.relation = derived_from`.
  Обратная петля: recall (B) пишет в `activation_log` → консолидация (C) подкрепляет `strength`.

**Статус в MVP.** Реализован поток B на вырожденных данных (символьный L1 поверх `facts.jsonl`,
флаг знакомости, habituation). Поток A тривиален (cue лежат в самом факте). Поток C (консолидация
L4/L5) — Приоритет 2, ещё нет.

## 4. Имена в коде

Три регистра, одна пара сущностей:
- **концептуальный** (психо-аналогия) — «cue-побудитель / familiarity» и «долговременный store / LTM»;
- **операционный** (код) — `cueMemory` (хэндл `cues`) и `factStore` (хэндл `facts`);
- **русский** (речь, доки) — **«зацепки»** и **«свод»**: «зацепка сработала по теме mysql → факт
  достали из свода». Читается в нужную сторону: зацепка цепляет → лезешь в свод за фактом. По дизайну
  «свод» не «срабатывает» (он пассивен) — срабатывает «зацепка». Альты: зацепки → наводки/маяки;
  свод → копилка/факты.

```js
cues.match(context)   // фаза-1: контекст × триггеры → взвод побуждения (флаг знакомости)
facts.get(hit)        // фаза-2: recollection — достаём содержимое
```

**Почему `cue`, а не `flash`/`working`.** Суть быстрого слоя — *побуждение* (priming), а не
«состояние сейчас» (working memory) и не «эфемерный лог» (flash). `cue` — родной термин концепции
(L0, `fact_cues`, `cue_json`), буквально «дистиллят триггеров», и поток `cue → fact` читается сам.
Альтернативы, если нужно подчеркнуть действие: `triggerMemory`/`triggers` (спусковой крючок,
ложится на «взведённый курок» болевой оси §10.3) или `primeMemory`/`prime` (психо-точное
«преднастройка к recall»). `coreMemory` для LTM **не берём** — в MemGPT/Letta это противоположный
(маленький in-context) блок; их аналог нашего LTM — `archival memory`.

## 5. Как употреблять в разработке

- **В коде/схеме:** `cueMemory` / `factStore` (хэндлы `cues` / `facts`). Носители: `cueMemory` =
  `fact_cues` + SQLite-зеркало + флаг (в MVP — поле `cues` в `facts.jsonl`); `factStore` =
  `facts`/`fact_*` (в MVP — `facts.jsonl`). `activation_log` — это **телеметрия консолидации**, не
  `cueMemory`.
- **«Побуждение / familiarity / фаза-1»** — про `cueMemory` (взвод, флаг знакомости).
  **«Recollection / фаза-2»** — про чтение `factStore`. **«Консолидация»** — только про поток C
  (L4+L5, наполнение store). Не смешивать три потока.
- **Не путать** `cueMemory` с контекстным окном LLM (это харнес) и с `activation_log` (телеметрия).
